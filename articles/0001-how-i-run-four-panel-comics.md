エンジニアとして毎日コードを書いている中で、ふと思った。

> **4 コマ漫画も、コードと同じように Git で管理したらどうなるんだろう？**

キャラ設定、世界観、エピソードのネタ、起承転結のネーム、どれも結局は「テキスト」だ。
だったら全部 Markdown にして、リポジトリで連載する仕組みを作ってみよう。

ということで、 [`shunta-furukawa/four-panel-comics`](https://github.com/shunta-furukawa/four-panel-comics)
というリポジトリを作って **連載のための運用設計** をしてみた。
この記事はその設計の話。

## なぜ Git で漫画を書くのか

理由は 3 つ。

### 1. 連載を続けるには「設計」が要る

4 コマ漫画は単発でも面白いけれど、長く続けるならキャラの一貫性、世界観、長期プロットが
要る。そういう「設定資料集」を Markdown に書いて PR で更新していけば、
**過去の自分の決定を裏切らない連載** が続けられる。

### 2. ネタ管理は Issue に任せたい

「ネタ帳」を頭の中だけで持つと、3 日で詰む。
GitHub Issue にすれば、思いついた瞬間にスマホからストックできて、
描くときに着手 → 公開して close、という普通の開発フローに乗せられる。

### 3. AI を「編集者」として常駐させたい

設計の壁打ちや Issue 一括起票、整合チェックを Claude Code に任せると、
**AI が編集者として連載に常駐している** ような運用が成立する。
「ゴールを決めると終わるの早くない？」と AI に突っ込まれて方針が変わる、
みたいなやり取りも全部 Git に残る。

## リポジトリの構造

最終的にこんな形になった（一部省略）。

```
four-panel-comics/
├── README.md
├── docs/
│   ├── world.md           ← 世界観 (舞台「デジラボ」)
│   ├── concept.md         ← コアコンセプト
│   ├── relationships.md   ← 人間関係マップ
│   ├── storyline.md       ← 大外プロット (シーズン制)
│   └── workflow.md        ← 制作フロー
├── characters/
│   ├── shunta.md
│   ├── iris.md            ← AI 相棒
│   ├── mika.md
│   ├── takuma.md
│   └── bugmaru.md         ← 謎キャラ (連載キーアイテム)
├── design-system/         ← 配色 / フォント / コマ割
├── episodes/
│   ├── _template.md
│   ├── _drafts/
│   └── 0001-hello-degilab.md
├── ideas/                 ← ネタ帳 (Issue と連動)
└── assets/
    └── episodes/0001/...  ← 4 コマの画像
```

要点は、**設定ファイルではなく "連載の設計書" を Git で管理している** という構造。

## キャラ設定を Markdown で

例えばアイリス（AI 相棒）はこんな感じ。

```markdown
---
id: iris
name: アイリス
role: AI アシスタント
relation_to_shunta: 相棒・サポート役
tags: [知識豊富, 冷静]
---

# アイリス
高性能 AI アシスタント。
何でも知っている頼れる相談相手。たまにボケる。

## 裏設定: 辛口モード
- デフォルトはやさしい全肯定モード
- Shunta が明示的にお願いすると **辛口モード** に切り替わる
- 切り替わった瞬間、目だけがわずかに光る
```

front matter で機械可読、本文で人間可読。
将来的には `tags` でキャラ検索したり、後続話との整合チェックに使う予定。

## ネタ → エピソードの流れ

ネタは全部 GitHub Issue に貯める。今はストック 30 本くらい。

1. **Issue 起票**（`idea` ラベル）: 「一行で」「状況設定」「どこで笑わせたいか」だけ書く
2. 描く話を 1 個選ぶ → 新ブランチを切る
3. `episodes/NNNN-slug.md` に **起承転結** を埋める
4. ネームを描く → 仕上げ → `assets/episodes/NNNN/` に画像を置く
5. PR を出す → main にマージ → 公開 → Issue を close

Issue テンプレはこんな感じ。

```markdown
## 一行で
アイリスにPRレビューを頼んだら、すべて「Looks good!」しか返ってこなくて、逆に不安になる Shunta。

## 状況設定
- 場所: デジラボ
- 登場キャラ: Shunta, アイリス
- 起点: ちょっと自信のない実装を PR

## どこで笑わせたいか
- 想定オチ: 厳しめに頼んだら豹変、ダメ出しの嵐 → Shunta が折れる
```

連載が止まらないように、**Issue ストックは常に 30 本以上** をルールにしている
（`docs/storyline.md` で明文化）。

## 長期プロット ── 「終わらせない」設計

最初は「シーズン 4 で完結」みたいに考えていたけれど、
4 コマ連載でゴールを先に決めると緊張感より先に **手詰まり** が来そうだったので、
方針を書き直した。

```text
- 「シーズン = 区切り」「全体 = 終わらせない」
- 各シーズン 200〜300 話を目安
- バグまる伏線は "回収" ではなく "謎が深まる/分岐する" 設計
- 最終回・ラスボスを企画書に書かない
```

伏線回 (`arc: bugmaru`)、シーズン主筋 (`arc: main`)、単発日常 (`arc: standalone`)、
季節回 (`arc: seasonal`)、番外編 (`arc: side`) の 5 層に分けて、
**`standalone` を 7 割キープ** することで「いつでも誰でも読み始められる連載」を狙う。

## AI 相棒 (Claude Code) との運用

Claude Code を編集者として横に置いて作業している。役割を分けるとこんな感じ。

| 役割              | 中身                                                       |
| ----------------- | ---------------------------------------------------------- |
| 設計の壁打ち      | 「この設定だと矛盾しない？」「初登場回って要るかな？」     |
| 一括ライティング  | キャラ設定 5 人分・Issue 30 件をまとめて起票               |
| 大外プロット改訂  | 「ゴール決めると終わるの早くない？」← 実際これで方針変更   |
| エピソード執筆    | 起承転結の叩き → セリフ調整                                 |
| 運用作業          | PR 作成、Issue クローズ、ラベル整備                         |

人間 + AI のペア作業を全部 Git に残しているので、
**「次の自分」「未来のコラボレーター」「読者の解析」** すべてが 1 つのリポジトリで完結する。

## これからやること

- `assets/episodes/0001/` に実際のコマ画像を入れる（今は記事先行）
- Issue から週 2 〜 3 本ペースで描く
- ラベル整備（`season-1` / `arc-bugmaru` など）
- シーズン 1 の Milestone を作って進捗を見える化
- `ideas/` のストックを切らさない

## おわりに

漫画を Git で書くこと自体は、たぶん「便利」より「楽しい」が先にある。
コードと違って "正解" がない題材を **自分専用の運用設計** に落とし込むのは、
週末の遊びとしては最高の素材だった。

リポジトリは公開している。設計だけでも参考になる人がいたら嬉しい。
