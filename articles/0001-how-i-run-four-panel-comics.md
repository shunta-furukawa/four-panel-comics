> 同時公開: 4 コマ漫画「はじめてのデジラボ」（記事冒頭の画像が第 1 話）です。

最初にお断りなのですが、これは **個人的な実験企画** のメモで、所属している会社の業務とは一切関係ありません。個人で契約している Claude Code のアカウントを使って、休みの日に趣味として触っているだけの内容です。

技術にも創作にも、ネット上の皆さんの方がずっとお詳しいと思うので、本記事は答えを示すものでは全然なくて、「自分はとりあえずこういう構えで試してみました、もし良いやり方をご存知でしたら教えてください」くらいのスタンスで読んでいただけたら嬉しいです。誤りや見落としはたくさんあると思います。

## なぜ「漫画」で生成 AI を試したかったのか

生成 AI は、システム開発のような **正解のあるカチッとした作業** で活躍するイメージが強いように思います。一方で、コンテンツやクリエイティブな創作活動にも、もしかしたら活かせる余地があるんじゃないかな、というのが今回のスタートでした。

特に物語をつくる場面で、生成 AI に任せたい役割を整理してみると、こんなことができそうな気がしています（あくまで現時点の仮説で、実際にやってみないと分からないところも多いです）。

- **整合性のガード役**: 過去話と矛盾する展開を、読者をがっかりさせる前にそっと指摘してくれる
- **壁打ち相手**: 「この展開だと、このキャラの性格とちょっとブレませんか？」を素早く返してくれる
- **設定・世界観の保持**: 何ヶ月後でも初期設定を覚えていてくれる
- **過去話との照合と補正**: 「以前これをこう描いたので、今回はこうしておくと自然かもしれません」と提案してくれる

漫画は、現実とは別の世界で別のキャラが生きている営みだと思っているので、その **世界法則を司る存在 / 番人** として生成 AI を置けたら面白いんじゃないか、というのが一番試したかったことです。「作者の頭の中だけにあった設定」を、共有できる形で誰か（AI）にも持ってもらえる、というイメージです。

## 仮説: 事前に「設計」を詰めておくとまとまりが出るのでは

もう一つ、運用設計を始めるにあたって立てた仮説があります。

> デザインシステム / キャラ設定 / 大枠のプロット / 何を描きたいか / なぜ描きたいか
> ── これらを **先に**書面で詰めておくと、まとまりのあるコンテンツが作りやすくなるのではないか。

これも本当にそうかは正直まだ分かりません。ただ、コードと同じで「あとから直す」のは大変だな、という感覚があり、特に漫画の場合は「あとから初期設定を直す」がもっと大変な気がしたので、最初にきちんと書面に落としておくことを試しています。間違っていたら、また直せばいいかなとも思っています。

## なぜ Git で書くか

理由はだいたい 3 つくらいです。

### 1. 設定の改訂を履歴で残したい

キャラの性格や世界観は連載中に「もう少しこういう人にしたい」と変わっていくものだと思います。その変遷を追えるようにしておくと、過去の自分の決定を「裏切らない」連載にしやすいかもしれない、と考えました。

### 2. ネタ管理を Issue に乗せたい

日常で思いついた小ネタをスマホからすぐ Issue に投げて、描くときにそれを引いてくる、というやり方ができれば、ネタ切れを少しは遅らせられるかもしれません。

### 3. AI に「編集者」として常駐してもらいたい

Claude Code が常にリポジトリの全体を見ている状態を作っておければ、上で書いた "世界法則の番人" としての役割が自然に回ってくれるんじゃないか、というのが期待しているところです。

ポイントは、コードではなく **連載のための設計書** を Git に置いている、というところかなと思っています。

## キャラ設定を Markdown で

例えば AI 相棒の「アイリス」は、こんな感じで書いています。

```markdown
---
id: iris
name: アイリス
role: AI アシスタント
relation_to_shunta: 相棒・サポート役
tags: [知識豊富, 冷静]
---

# アイリス
高性能 AI アシスタント。
何でも知っている頼れる相談相手。たまにボケる。

## 裏設定: 辛口モード
- デフォルトはやさしい全肯定モード
- Shunta が明示的にお願いすると **辛口モード** に切り替わる
- 切り替わった瞬間、目だけがわずかに光る
```

front matter は機械が読みやすく、本文は人間が読みやすく、という雑な使い分けです。ゆくゆくは tags でキャラ検索したり、後続話との整合チェックに使ってみたいなと思っています（このあたりはまだ手付かずで、うまく回るか分かりません）。

## ネタ → エピソードの流れ

ネタは全部 GitHub Issue に貯めるようにしてみました。

1. **Issue 起票**（`idea` ラベル）: 「一行で」「状況設定」「どこで笑わせたいか」だけ書く
2. 描く話を 1 個選ぶ → 新しいブランチを切る
3. `episodes/NNNN-slug.md` に **起承転結** を埋める
4. ネームを描く → 仕上げ → `assets/episodes/NNNN/` に画像を置く
5. PR を出す → main にマージ → 公開 → Issue を close

Issue のテンプレはこんな感じにしています。

```markdown
## 一行で
アイリスにPRレビューを頼んだら、すべて「Looks good!」しか返ってこなくて、逆に不安になる Shunta。

## 状況設定
- 場所: デジラボ
- 登場キャラ: Shunta, アイリス
- 起点: ちょっと自信のない実装を PR

## どこで笑わせたいか
- 想定オチ: 厳しめに頼んだら豹変、ダメ出しの嵐 → Shunta が折れる
```

ネタ切れが怖いので、ストックは常に 30 本以上を目安にしています（とはいえ本当に切れない保証はないので、ここは様子を見つつ調整するつもりです）。

## 大枠のプロット ── 「終わらせない」設計

最初は「シーズン 4 で完結」のようなゴール付きの構造を考えていました。が、4 コマ連載でゴールを先に決めてしまうと、緊張感よりも先に手詰まりが来てしまいそうだなと思って、書き直しました。

```text
- 「シーズン = 区切り」「全体 = 終わらせない」
- 各シーズン 200〜300 話を目安
- 伏線は "回収" ではなく "謎が深まる/分岐する" 設計
- 最終回・ラスボスを企画書に書かない
```

これも本当に長く続けられるかは未知数ですが、当面はこの構えで進めてみるつもりです。続けてみてダメだったら、また方針を書き直そうと思います。

## AI 相棒（Claude Code）にお願いしていること

「世界法則の番人」と言いつつ、実際にお願いしている作業はもう少し地味です。今のところはこんな感じ。

| 役割              | 中身                                                       |
| ----------------- | ---------------------------------------------------------- |
| 整合性のガード    | 「この設定、前のキャラ像とちょっとブレませんか？」と聞く   |
| 壁打ち相手        | 「初登場回って要りますかね？」と相談したら方針が変わった   |
| 設定の編集        | キャラ設定書のリファクタや表記ゆれの統一                    |
| 一括ライティング  | あるあるネタを 10 〜 20 本まとめて Issue に起こしてもらう  |
| 大枠のレビュー    | 「ゴール決めると終わるの早くないですか？」と突っ込まれて、長期連載仕様に書き直した |

人間（自分）と AI の往復のやり取りを全部 Git に残せるので、あとから「なぜこの方針にしたんだっけ」を追えるのも、地味にありがたいなと感じています。

## 第 1 話「はじめてのデジラボ」

Shunta が「毎日 4 コマ描こう」と決めたものの、結局 README ばかり書いてしまうメタな起点回にしました。半分以上、自分で実際にやってしまったやつです。

## 公開のペースについて

毎日連載はしないことにしました。続かないのが目に見えていたのと、思想・テーマ系の話は **記事 1 本＋関連 4 コマ数本** をひとまとめにした「テーマリリース」、軽いあるあるネタは **4 コマ単独** の「単発リリース」、という 2 種類のリリース単位を行き来する方が、自分には現実的そうかなと思っています。ここも仮説なので、運用しながら見直すつもりです。

繰り返しになりますが、本記事の取り組みは **個人的な実験企画** です。所属している会社の業務とは一切関係なく、個人で契約している Claude Code のアカウントを使って、趣味として触っているだけの試行錯誤です。

技術的にも創作的にも、皆さんの方がずっとお詳しいと思うので、「こうした方がいいよ」「自分ならこうしている」といったヒントやツッコミをいただけると、本当に勉強になります。
